# P10 — Session Generation

## Overview
A session fixation vulnerability arises when the application fails to **rotate the session identifier** at any transition in privilege level — most critically at login, but also on privilege escalation (user → admin), logout, or re-authentication. If the pre-authentication session ID is preserved, an attacker who obtained or planted that ID (via a crafted link, subdomain cookie injection, or XSS) rides the same ID into the authenticated session and becomes the victim without ever knowing their password. The root cause is always the same: the code mutates session state (`userId`, roles, `auth = true`) on the *existing* session object instead of discarding it and issuing a fresh, unpredictable ID. Stateless JWT bearer flows where a brand-new token is minted per login are out of scope here (see P14-JWTValidation).

## What to check
- On successful login, is a **new** session ID issued and the old one invalidated, before any authenticated state (`userId`, roles) is attached?
- Is regeneration enforced at every privilege transition: anonymous → authenticated, user → admin/impersonation, and on re-authentication (e.g. step-up)?
- Is the old session destroyed server-side (not just overwritten) so its ID can no longer be replayed?
- Does the session cookie carry the `HttpOnly`, `Secure`, and `SameSite` attributes, and is the value generated by a CSPRNG (≥128 bits of entropy)? Low-entropy or sequential IDs enable brute-forcing regardless of fixation.
- Is the session ID ever accepted from the URL (query string / path) or a non-cookie source, exposing it to referrer/SSN leakage?
- On logout, is the session invalidated **and** a new empty ID issued, preventing reuse of the post-logout ID?

## Static signals
Missing regeneration at login (Node/Express):
- `req.session.userId = user.id;` with no preceding `req.session.regenerate(...)` / `regenerate()`
- `ctx.session.uid = user.id` (koa-session) with no `ctx.session` re-assignment
- Assigning onto the same session object inside the login branch and returning the same `Set-Cookie`

Present-but-misused APIs (regeneration called but ignored):
- `req.session.regenerate(err => { ... })` where the auth state is attached *outside* the callback (race / no-op)
- `req.session.regenerate()` awaited but old cookie still sent because the response writes headers before regeneration completes

Other languages/frameworks:
- Python: Flask — assigning `session['user_id'] = user.id` without `session.clear()` then `session` re-issue; Django — not using `django.contrib.auth.login(request, user)` (which rotates) but manually setting `request.session` keys
- Java: Servlet — calling `request.changeSessionId()` (Servlet 3.1+) is the fix; absence + `session.setAttribute("user", ...)` is the smell. Spring Security rotates by default — confirm `sessionManagement().sessionFixation()` wasn't set to `none`
- Go: `gorilla/sessions` — `session.Values["uid"] = id` without `sessions.NewSession`/re-save; or `gin` `c.Set("user", ...)` on the same session
- PHP: relying on `session_regenerate_id()` being absent; presence of `session_regenerate_id(true)` is the safe pattern
- Ruby/Rails: Devise/`reset_session` — check `reset_session` is invoked before `session[:user_id]=`; bare `session[:user_id] = user.id` is the smell

Token/JTI reuse across privilege levels:
- Same JWT `jti` / session ID retained after `roles: ['admin']` is appended to the payload
- Refresh-token rotation skipped when scopes are elevated

## False positives
- Stateless API using per-request JWTs minted fresh on each login (no server-side session object) — covered by P14-JWTValidation, not this perspective.
- Framework auto-rotates on authentication: Express-session + Passport, Spring Security's default `changeSessionId`, ASP.NET Core cookie auth (`CookieAuthenticationEvents`), Django `auth.login`. Confirm the default wasn't disabled.
- Single-page apps where the only "session" is an opaque bearer token issued *after* login and no pre-auth session exists — there is nothing to fixate.
- Read-only public pages with no session creation before login.

## Attack scenario
1. Attacker obtains a valid anonymous session ID from `app.example.com` (e.g. by visiting the site, or via an XSS/Subdomain that can set the cookie).
2. Attacker injects/fixes that ID onto the victim's browser — a link `https://app.example.com/login?SID=ATTACKER_ID` if the app reads SID from the query, or a cookie set from a vulnerable sibling subdomain.
3. Victim authenticates normally; the server attaches `userId`, roles, etc. to the **same** session ID it never rotated.
4. Attacker presents the known ID and is now authenticated as the victim — full account takeover, no credentials needed.
5. If the victim escalates to admin (and the ID still isn't rotated), the attacker inherits admin privileges.

## Impact
- **Confidentiality**: full access to the victim's account data and any resource their role can read.
- **Integrity**: actions performed as the victim (transfers, config changes, admin operations) — account takeover.
- **Availability**: account lockout/deletion, mass session invalidation if escalated to admin.
- Severity scales with the compromised identity: a fixated admin session yields full application compromise. Pre-auth fixation bypasses password/MFA entirely.

## Remediation
Rotate the session ID at login (and any privilege change) **before** binding identity to it:
```ts
// VULNERABLE — pre-auth session ID is kept; attacker who knows it is now logged in
req.session.userId = user.id;
req.session.roles = user.roles;
res.redirect('/dashboard');

// SAFE — discard the old session, issue a new ID, then attach identity
req.session.regenerate((err) => {
  if (err) return next(err);
  req.session.userId = user.id;
  req.session.roles = user.roles;
  req.session.createdAt = Date.now();
  res.redirect('/dashboard');
});
```
For Servlet containers use `request.changeSessionId()`; in PHP `session_regenerate_id(true)`; in Rails call `reset_session` before `session[:user_id]=`. Defense-in-depth: set `HttpOnly; Secure; SameSite=Lax` (or `Strict`), bind the session to a server-side timeout, and short-rotate on step-up authentication.
