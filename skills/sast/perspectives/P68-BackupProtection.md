# P68 — Backup Protection

## Preconditions

The code creates backups.


## Overview
Backups, database dumps, snapshots, and export files typically contain the **complete** production dataset — often more than any single authenticated user can reach through the application. When those artifacts are stored unencrypted, written to a world-readable location, protected by weaker access controls than production, or generated by an export endpoint without authorization, they become a high-value shortcut for an attacker: one leaked object yields full data exfiltration without touching the application at all. The root cause is almost always a **protection asymmetry** — production is hardened (TLS, RBAC, encryption at rest, audit logging) while the backups produced from it are treated as second-class and left plaintext in a public bucket, a web root, or an unauthenticated download path.

## What to check
- Are DB dumps / snapshots / exports **encrypted at rest** (server-side envelope or object-storage SSE with a customer-managed key), or written as plaintext files?
- Is the backup destination (S3/GCS/Azure Blob bucket, NFS share, on-disk path) **private by default** with least-privilege IAM — no `public-read`, no overly broad `s3:*` policy, no wildcard principal?
- Do export/download endpoints (`/admin/export`, `/backup.sql`, `/api/v1/users.csv`, `.git/`, `~`, `.bak`) enforce the **same authorization** as the source data (RBAC/ABAC, session, tenant scoping)? Cross-check with P24–P32.
- Is a **separate key** used for backup encryption (not the production data-encryption key), and is it stored outside application config / not in the same KMS as the data?
- Are backup retention and lifecycle configured so artifacts are not immortal in a forgotten location, and is deletion tied to a compliance window?
- Does the app or CI/CD write transient dumps (`*.sql`, `*.dump`, `*.bak`, `db.sqlite3`, `~`) into a **web-served directory** or a path that ends up in version control?
- Are secrets (DB credentials, API keys) embedded in the dump/snapshot, and would restoring it leak them?
- Is there logging/monitoring on backup read access (detection), not just write?

## Static signals
Plaintext dumps written to disk or object storage without encryption:
- `pg_dump $DB > backup.sql` / `mysqldump ... > dump.sql` / `sqlite3 db .dump > dump.sql` piped straight to a file
- `aws s3 cp backup.sql s3://bucket/` / `gsutil cp dump.sql gs://bucket/` / `az storage blob upload` with no `--sse` / KMS key id
- Python: `shutil.copy(db_path, '/var/backups/db.sqlite3')`, `open('backup.json','w').write(json.dumps(rows))`
- Node: `fs.writeFileSync('backup.json', JSON.stringify(db))`, `res.download('backup.sql')`
- Ruby: `File.write('dump.sql', ActiveRecord::Base.connection.execute('pg_dump ...'))`

Overly permissive / public destination config:
- Terraform: `aws_s3_bucket_acl { acl = "public-read" }`, missing `aws_s3_bucket_public_access_block`, `aws_s3_bucket_policy` with `Principal: "*"`
- CloudFormation / CDK: `Bucket(..., blockPublicAccess=BlockPublicAccess.BLOCK_NONE)`
- `gsutil acl ch -u AllUsers:R gs://bucket` / `gsutil iam ch allUsers:objectViewer`
- IAM policy with `"Action": "s3:*", "Principal": "*"` or `Resource: "*"`

Unauthenticated or under-authorized export endpoints:
- `@app.route('/export')` / `app.get('/export', ...)` with **no auth middleware** (no `@login_required`, no `requireAuth`, no guard)
- `if (user) { return res.download(backupPath) }` — present-but-unscoped (any logged-in user, no role/tenant check)
- Static-file serving rooted at a directory containing dumps: `app.use(express.static(__dirname))`, `app.use('/backups', express.static(...))`, Django `STATICFILES_DIRS` including a backup dir
- Web server aliases: nginx `location /backup { alias /var/backups/; }`, Apache `Alias /backup /var/backups`

Backup key == production key / secret in config:
- `KMS_KEY_ID` reused for both app data and backups
- `DB_PASSWORD`, `BACKUP_PASSPHRASE` hardcoded in shell scripts or checked into `.env` / repo

## False positives
- Object storage uses **SSE-S3 or SSE-KMS with a customer-managed key**, has a `public_access_block` (all four blocks on) and a least-privilege bucket policy — backups are protected.
- The export endpoint sits behind the same auth/RBAC/tenant guard as the underlying data and scopes the query to the caller (e.g., `WHERE tenant_id = current_tenant`).
- The "backup" is a **generated-on-demand**, time-limited signed URL (STS presigned with short expiry) rather than a long-lived public object.
- This is mostly an **operational/config concern** not visible in application source. If IaC (Terraform/CloudFormation), bucket policy, or CI scripts are unavailable, record it as **Low / Informational — cannot confirm from source** and flag it for ops review rather than asserting a vuln.

## Attack scenario
1. Attacker enumerates object storage (leaked bucket name, `s3://company-backups`, DNS, GitHub history) or guesses an export path (`/backup.sql`, `/db.bak`, `/.git/config`).
2. The destination is `public-read` (or has a wildcard policy) and the dump is unencrypted — attacker runs `aws s3 cp s3://company-backups/prod-2026-06-30.sql -` (no credentials) or simply `curl https://app.example.com/backup.sql`.
3. The dump contains the full `users` table: password hashes, emails, PII, and often service-account tokens embedded in config tables.
4. Attacker cracks the hashes offline, pivots into user accounts, and sells/exfiltrates the PII — all without ever sending a request to the hardened application. Because production access logs were never touched, the breach can go undetected for months.

## Impact
- **Confidentiality**: total dataset disclosure — typically far exceeding what any single endpoint exposes. PII, credentials, secrets, and tokens are all in scope.
- **Integrity**: a forged/restored backup can be used to poison production state (ransomware-style overwrite).
- **Availability**: deletion or ransomware-encryption of the only backups destroys recoverability after a primary incident.
- Severity scales with dataset sensitivity and exposure width: a **public, unencrypted** full-DB dump is **Critical**; a **plaintext dump on an internal-only host** reachable by a broad internal principal is **High**; an authenticated export missing tenant scoping is **High** (IDOR-flavored). Confirm from config before rating.

## Remediation
Encrypt with a separate key, store privately, and scope export endpoints to the caller:
```bash
# VULNERABLE — plaintext dump pushed to a public bucket
pg_dump "$DB" > backup.sql
aws s3 cp backup.sql s3://company-backups/   # bucket is public-read, no SSE

# SAFE — encrypted at rest with a separate KMS key, private bucket, least privilege
pg_dump "$DB" | openssl enc -aes-256-gcm -pbkdf2 -pass file:/run/secrets/backup_key \
  | aws s3 cp - "s3://company-backups/$(date +%F).sql.enc" \
      --sse aws:kms --sse-kms-key-id "$BACKUP_KMS_KEY_ID"
# plus: public_access_block (all 4), bucket policy restricted to the backup role,
#       retention lifecycle, and CloudTrail/S3 access logging on read.
```
```js
// VULNERABLE — any logged-in user downloads the full DB dump
app.get('/export', (req, res) => res.download('/var/backups/prod.sql'));

// SAFE — admin-only, tenant-scoped, streaming, never touches a web-served dir
app.get('/export',
  requireRole('admin'),
  auditLog('export'),
  async (req, res) => {
    const rows = await db.query('SELECT * FROM records WHERE tenant_id=$1', [req.user.tenantId]);
    res.setHeader('Content-Disposition', `attachment; filename="export-${Date.now()}.csv"`);
    csvStream(rows).pipe(res);   // generated in-memory, no on-disk artifact
  });
```
Defense-in-depth: keep backup encryption keys in a separate KMS from production data keys, enforce MFA-delete + versioning + object-lock on the backup bucket, and alert on any anonymous or out-of-role read.
