# P16 — SessionIDEntropy

## Preconditions

The code generates session identifiers.


## Overview
A session identifier is the sole proof of identity for an authenticated user after login; if it is predictable, an attacker can forge a valid session without ever knowing the password. The root cause is almost always **home-grown ID generation** that derives the token from a low-entropy or guessable input — a sequential counter, a timestamp, a username, or a weak PRNG such as `Math.random()` / `java.util.Random`. Even an 128-bit ID is worthless if half of the bits are constant or derived from public data. The fix is to delegate to a cryptographically secure random number generator (CSPRNG) emitting at least 128 bits of entropy, or — far better — to use the framework's hardened session store, which already does this and rotates, signs, and expires tokens correctly.

## What to check
- Is any session ID, session token, remember-me token, password-reset token, or "magic link" generated by application code rather than the framework/session library?
- Is the value produced by a **non-cryptographic** RNG — `Math.random`, `Math.random().toString(36)`, `java.util.Random`, Python's default `random` module, PHP `rand()`/`mt_rand()`, Ruby `rand`, Go `math/rand`, .NET `System.Random`?
- Is the ID derived from deterministic or publicly knowable inputs — sequential DB id, `Date.now()`/`gettimeofday`, username/email, IP address, PID, hostname, `uuid v1` (MAC + clock), or a plain hash of any of these?
- Is the **effective entropy** below 128 bits? Count only random bits: `user.id + '-' + Date.now()` has ~20 bits of guessability; `md5(user.email)` has zero (the input is public).
- Is the ID space small enough to enumerate (short numeric token, 4–6 digit PIN-style session, base36 ≤ 8 chars)?
- Are tokens logged, cached in plaintext, or embedded in URLs (query string) where they leak via referrers/proxies/logs?
- After login or privilege change, is a **new** ID issued (rotation)? Reusing the pre-login ID invites session fixation.
- For JWT/stateless sessions: is the signature key (`HS256` secret or private key) strong and rotated, and is a `jti` present with CSPRNG entropy if revocation is expected?

## Static signals
Weak RNG feeding token generation:
- Node: `Math.random().toString(36)`, `crypto.createHash('md5').update(...)` for a session id
- Python: `import random; token = ''.join(random.choices(...))`, `secrets` *not* used; `uuid.uuid1()` (time+MAC)
- Java: `new Random().nextLong()`, `Math.random()`, `Integer.toString(ThreadLocalRandom.current().nextInt())`
- Go: `math/rand` (`rand.Int63()`, `rand.Read`) instead of `crypto/rand`
- PHP: `rand()`, `mt_rand()`, `uniqid()` (no `more_entropy`), `md5(uniqid())`
- Ruby: `rand`, `SecureRandom` *not* used; `Digest::MD5.hexdigest(...)` for tokens
- .NET: `new Random().Next()`, `Guid.NewGuid()` *can* be OK but is not guaranteed cryptographic

Deterministic / low-entropy derivation:
- `sid = user.id + '-' + Date.now()`
- `sid = sha256(username + '|' + password)` / `md5(email)`
- `token = str(User.objects.count())` / `SELECT MAX(id)+1`
- `sid = base36(timestamp)[2:]` (truncated, small space)
- Token embedded in URL: `res.redirect('/app?session=' + sid)`, `<a href="...?token=...">`

Safe patterns (confirm these are present, not the above):
- Node: `crypto.randomBytes(32).toString('base64url')`, or delegate to `express-session`, `next-auth`, `@nestjs/cookie-session`
- Python: `secrets.token_urlsafe(32)`, Django/Flask default session
- Java: `SecureRandom.getSeed(32)`, Spring Session default
- Go: `crypto/rand` (`rand.Read`, `hex.EncodeToString`)
- PHP: `random_bytes(32)` / `bin2hex`, Laravel/Symfony default session
- Ruby: `SecureRandom.hex(32)` / Rails `session`

## False positives
- The application uses a mainstream framework session store with defaults (`express-session`, `next-auth`, Django/Flask sessions, Spring Session, Rails `session`, Laravel/Symfony session) — these use CSPRNG with ≥128 bits and sign cookies. Confirm the secret/encryption key is strong and not committed to source.
- Stateless JWTs where security rests entirely on the HMAC/RSA signature (no session ID to guess) — verify the signing key strength instead; this perspective does not apply to the token value itself.
- Short numeric values that are **not** bearer tokens — e.g. an opaque internal row id echoed in a URL but gated server-side by authorization checks (still a concern under IDOR, not here).
- Test/dev codepaths guarded behind `NODE_ENV === 'test'` or a feature flag that never ships to production.

## Attack scenario
1. The app sets `sid = user_id + '-' + Date.now()`, e.g. `1042-1719800000000`.
2. Attacker registers a normal account, observes their own `sid`, and learns the `user_id` of an admin (often exposed in public profile URLs).
3. The timestamp narrows to a few thousand candidate values around login time; the attacker scripts login attempts with `1043-<ts>` for each candidate millisecond.
4. One matches the admin's active session — no password required. The attacker now operates as admin.
5. With sequential `mt_rand()`-style tokens, the attacker computes the internal PRNG state from two observed IDs and predicts all future tokens for every user.

## Impact
- **Confidentiality**: full account takeover — the attacker reads all data accessible to the impersonated user.
- **Integrity**: actions performed as the victim (transfers, config changes, admin operations).
- **Availability**: mass session forgery against admin accounts can disable the service or wipe data.
- Severity scales with the victim's privileges and how quickly IDs rotate. Predictable admin tokens → critical; low-privilege with short TTL → high. A weak PRNG shared across all users is a single-attacker-takes-everyone scenario.

## Remediation
Delegate token generation to the framework, or generate with a CSPRNG and ≥128 bits of entropy:
```ts
// VULNERABLE — derived from public/sequential data
const sid = `${user.id}-${Date.now()}`;
const token = crypto.createHash('md5').update(user.email).digest('hex');

// SAFE — 256 bits of CSPRNG entropy, URL-safe
const sid = crypto.randomBytes(32).toString('base64url');
// or, preferred: let the framework handle it
app.use(session({
  genid: () => crypto.randomBytes(32).toString('base64url'),
  secret: process.env.SESSION_SECRET, // strong, rotated, not in source
  resave: false, saveUninitialized: false,
  cookie: { httpOnly: true, secure: true, sameSite: 'lax' },
}));
```
Rotate the ID on login and on any privilege change (defense-in-depth against fixation), set cookies `HttpOnly`, `Secure`, `SameSite`, and never place a bearer token in a URL query string.
